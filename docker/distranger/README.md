# DistRanger Docker deployment

This directory keeps the existing `dockerfile` and `docker-compose.yml`.
`deploy.sh` is an optional deployment entry point that does not modify the
Dockerfile. It can:

- Clone the DistRanger source tree when requested.
- Build the runtime image from `docker/distranger/dockerfile`.
- Start the container through Docker Compose.
- Mount the source directory at `/workspace/DistRanger`.
- Mount an output directory at `/workspace/output`.
- Mount `/var/run/docker.sock`, so DistRanger can access Docker-managed
  database containers on the host.

## Prerequisites

The host needs:

- Docker
- Docker Compose v2, available as `docker compose`
- Git, only when automatic cloning is used

## Run with the current source tree

From the repository root, run:

```bash
./docker/distranger/deploy.sh
```

Default behavior:

- Image name: `distranger:latest`
- Source mount: current repository to `/workspace/DistRanger`
- Output mount: `./docker-output` to `/workspace/output`
- Container command: `bash`

Inside the container, build and run manually:

```bash
bash compile.sh
```

You can also pass a command directly to the script:

```bash
./docker/distranger/deploy.sh bash main/citus/ddc_test.sh
```

## Clone from Git and start

```bash
DISTRANGER_REPO_URL=https://github.com/USER/DistRanger.git \
DISTRANGER_SOURCE_DIR=/tmp/DistRanger \
./docker/distranger/deploy.sh
```

Specify a branch or tag:

```bash
DISTRANGER_REPO_URL=https://github.com/USER/DistRanger.git \
DISTRANGER_SOURCE_DIR=/tmp/DistRanger \
DISTRANGER_BRANCH=main \
./docker/distranger/deploy.sh
```

If the source directory already exists, the script does not clone again.

## Common options

```bash
DISTRANGER_IMAGE=distranger:dev \
./docker/distranger/deploy.sh

DISTRANGER_NO_BUILD=1 ./docker/distranger/deploy.sh

DISTRANGER_OUTPUT_DIR=/tmp/distranger-output ./docker/distranger/deploy.sh

./docker/distranger/deploy.sh bash
```

## Database connectivity

The existing configuration files use `host.docker.internal` to reach database
ports exposed on the host, for example:

- `main/citus/citus_ddc_config.txt`
- `main/shardingsphere/ss_ddc_config.txt`
- `main/clickhouse/clickhouse_ddc_config.txt`
- `main/tidb/tidb_ddc_config.txt`
- `main/vitess/vitess_ddc_config.txt`

The Compose override generated by the script adds:

```yaml
extra_hosts:
  - "host.docker.internal:host-gateway"
```

This makes `host.docker.internal` resolve on Linux hosts. If the database
containers do not expose their ports on the host, update the matching
`*_ddc_config.txt` file first.
